# v10.6 — Forward-looking marker + throttle race fix

## Two issues fixed

### Issue 1: Marker missing while stationary at a stop (the bug)

**What was happening:** On opening a route panel, `startLiveTracking`
begins immediately, but `setActiveRoute` (which populates the stop
list for distance comparisons) only runs after the route data fetch
completes ~200ms later.

If a GPS fix arrived in that 200ms window, `handlePosition` ran with
an empty stop list, consumed the throttle budget (`lastUpdateAt = now`),
and early-returned. The subsequent `setActiveRoute → getCurrentPosition`
re-request was then blocked by the 3-second throttle. With the user
stationary, the next watch tick could be 30+ seconds away.

Result: you arrive at the stop, open the route panel, and the marker
never appears.

**Fix:** `setActiveRoute` now resets `lastUpdateAt = 0`, guaranteeing
the next position fix is processed immediately rather than blocked
by a stale throttle.

### Issue 2: Marker shows the previous stop, not the upcoming one

**What was happening:** The highlight logic always picked the
geographically closest stop. When between #43 and #44 with the bus
moving forward, you might still be slightly closer to #43, so the
marker showed #43 — even though you'd already passed it.

**Fix:** Track `lastVisitedSeq` monotonically. Once the user has been
within 80m of a stop (the "at" threshold), we record its sequence.
Subsequent positions show the NEXT stop in sequence (`visited + 1`)
when between stops, with label "Approaching".

The behaviour:

- **At a stop** (≤80m): "📍 You are here · At stop #44"
- **Between stops, after visiting one**: "📍 Approaching (180m) · Approaching stop #45"
- **Between stops, before visiting any** (boarded mid-route, no
  history yet): falls back to v10.5 nearest-stop behaviour
- **Drifted off route after a visit**: "Not on this route, closest stop #N"

`lastVisitedSeq` is monotonic — GPS jitter cannot cause it to go
backward. It resets only on direction switch, panel close, or fresh
panel open.

## Diagnostic logging

Added a `[liveloc]` log line per position fix summarising:
- GPS accuracy
- Nearest stop and its distance
- Highlight mode (at/approaching/off)
- Visited-seq history
- Final highlighted stop

If the marker fails to appear in the future, run
`tail -f ~/.sg-bus-tracker.log | grep liveloc` while reproducing — the
logs will pinpoint whether it's a GPS issue, a threshold issue, or
something else.

## Known residual: boarding mid-route

If you open the route panel while already on a moving bus and you've
never been within 80m of any stop on this route in this session, the
forward-looking logic has nothing to anchor on. It falls back to
distance-based nearest-stop highlighting (v10.5 behaviour), which can
show the previous stop briefly. As soon as you reach the next stop
(within 80m), the visited-seq updates and forward-looking takes over.

This case requires direction-of-travel inference to fix properly
— a larger change for a future version if it proves common in
practice.
