# v10.9 — Search reveals when results are truncated

## What changed

Searching common terms like "bedok" matches ~100+ stops, but v10.8
silently capped at 30. The user couldn't tell whether they'd seen
everything or whether the cap was hiding stops.

v10.9:

- Cap raised from 30 to **100** (covers virtually all real searches)
- When results are capped, a footer appears below the suggestions:
  "Showing 100 of 137 matches. Type more to refine."
- Performance is fine — linear scan over ~5500 stops is sub-millisecond

## Why 100 not unlimited

Tested: searching "rd" or "ave" matches several thousand stops.
Rendering thousands of suggestion items would freeze the UI on a
phone. 100 is a safety ceiling, with the footer as the honest signal
when truncation kicks in.

## Implementation

`findStops` now returns `{ items, totalMatches, truncated }` instead
of a raw array. Both call sites (main search bar, planner destination
picker) updated.

The footer uses `class="suggestion-truncated"` — a different class
from `suggestion-item`, so arrow-key navigation and click handlers
naturally ignore it.

## What you'll see

- Searching "blk 55" → 1 result, no footer
- Searching "tampines" → ~40 results, no footer (under cap)
- Searching "bedok" → 100 results + "Showing 100 of 137 matches"
- Searching "rd" → 100 results + "Showing 100 of N matches"
