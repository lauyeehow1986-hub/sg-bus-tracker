# v10.13 — Security: bind proxy to localhost only

## What was wrong

`proxy.py` bound to `0.0.0.0`, which means the proxy listened on
**all** network interfaces — loopback (127.0.0.1) AND the device's
LAN IP. Anyone on the same WiFi as your phone could hit:

- `/keycheck` — see if your LTA API key is set
- `/setkey?key=ATTACKER_KEY` — silently overwrite your LTA API key
- `/lta/...` — drain your LTA quota
- `/logs` — read your recent search/usage history
- `/log` (POST) — poison your client log buffer

This was meaningful exposure on public WiFi (cafes, hotels, MRT
Wireless@SG, etc). On home WiFi it was lower risk — limited to
people you already shared the network with — but still avoidable.

The bug was hiding in plain sight from v1. The CORS-`*` and
`Access-Control-Allow-Private-Network: true` headers (originally
added to support a "fetch the page from LAN IP" flow) compounded
the issue by telling browsers to allow cross-origin LAN access.

## The fix

Single-line change in `proxy.py`:

```python
# OLD
server = ThreadingHTTPServer(("0.0.0.0", PROXY_PORT), ProxyHandler)

# NEW
server = ThreadingHTTPServer(("127.0.0.1", PROXY_PORT), ProxyHandler)
```

Verified locally that after the change:
- `127.0.0.1:8080` accepts connections (your normal usage works)
- `<external-IP>:8080` refuses connections (LAN attack surface gone)

## What you lose

The ability to access the proxy from another device on the same
network (e.g. testing from a laptop while the proxy runs on the
phone). The original setup printed both 127.0.0.1 and LAN IP URLs
on startup — only 127.0.0.1 works now.

If you ever need laptop access, ADB port forwarding or USB tether
both work without re-exposing the LAN attack surface.

## Comments cleaned up

- Console output now says "Bind: 127.0.0.1:8080 (localhost only)"
  instead of printing both URLs
- "Permissive CORS for LAN use only" comment replaced with a note
  explaining the headers are now defensive (no remote origin can
  reach the server in the first place)
- "*** THE KEY CHROME FIX ***" comment replaced with a note that
  the PNA header is no longer load-bearing (kept for compatibility,
  but same-origin requests don't trigger Chrome's PNA check)

The CORS-`*` and PNA headers are kept rather than removed — they're
now harmless (because of the bind) and removing them would require
more verification that nothing depends on them. The bind is the
load-bearing fix.

## Other security findings deliberately not addressed

These exist but are now lower-priority since the bind eliminates the
remote attack vector:

- `/setkey` accepts the API key as a query parameter (logged in
  proxy stdout). With localhost-only access, only processes on your
  phone can hit it, and they can already read the key file.
- No request body size limit on `/log`. Same reasoning.
- Path-traversal `startswith` edge case in `serve_static`. Same.
- Unbounded cache size. Same.

If you ever switch the bind back to LAN access (e.g. for laptop
testing), revisit these.
