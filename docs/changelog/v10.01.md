# v10.1 — Transfer alternatives always shown

## The problem

The planner previously stopped searching once it found a direct bus.
But a direct bus isn't always the fastest — sometimes a quick transfer
gets you there sooner. Google Maps shows all competitive options
side-by-side; the app was hiding them.

## The change

After finding direct bus options, the planner now also searches for
1-transfer alternatives in the background, and appends any that are
time-competitive (within +8 min of the fastest direct, max 3 shown).

## UI flow

1. User taps "Go"
2. Direct bus options render instantly (unchanged)
3. Below them, a small loader appears: "Checking for faster transfer
   options…"
4. Background search finishes in ~1-3 seconds
5. Loader is replaced with either:
   - An "ALSO CONSIDER · N transfer options" section, or
   - (silently removed, if no competitive transfers exist)

## Why background not blocking

Transfer search requires fetching route data for each service at origin.
On first use of a new origin stop, this can take ~2 seconds. Blocking
would make the user wait for results they may not even want. Rendering
direct options first then filling in alternatives feels snappier.

Abort-safe: if the user navigates away mid-search, the render is
abandoned, no stale content gets injected.

## Filter rationale

Threshold +8 min from fastest direct, max 3 shown. Matches Google
Maps' instinct — they hide 40-min transfers when 18-min direct
exists, but show 17-24 min alternatives. The +8 keeps the list
focused on genuinely competitive routes.

## Caveats

Transfer time estimates use `estimateTravelMins(stops, distKm)` plus
10 min transfer wait. These are rough heuristics, not live data.
LTA's API doesn't expose reliable live tracking for the *connecting*
bus. Expect the time numbers to be accurate ±20% for planning, not
minute-exact.
