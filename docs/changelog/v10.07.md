# v10.7 — Direction-of-travel fix for marker

## The bug (from user screenshot)

The screenshot showed the marker stuck on "Approaching Bedok Camp 2"
(stop #15, 173m away) — but the user's GPS dot was clearly past Bedok
Camp 2 on Upp East Coast Rd. The next stops on the route (#16 Opp Kew
Green Condo, #17 Temasek Sec Sch) should have been highlighted.

## Why it happened

The v10.6 forward-looking logic activates only after the user has been
within 80m of *any* stop on this route in the current panel session.
When you open the panel while already on a moving bus and don't pass
within 80m of any stop's exact coordinates (which is typical — the bus
drives past stops at 5-15m offset, and GPS error of 30-50m means we
rarely hit the 80m threshold while moving), the forward-looking model
never anchors. The pre-history fallback was just "highlight the nearest
stop" — which can be a stop you've already passed.

## The fix

Added direction-of-travel inference using two consecutive GPS fixes.

For each candidate stop within 250m of the current position, compare:
- Distance from the previous GPS fix
- Distance from the current GPS fix

If you're getting closer to the stop, it's ahead of you. The algorithm
picks the lowest-sequence stop among ahead stops — that's the next one
you'll reach.

Once an "ahead" stop is identified, the previous-sequence stop is
retroactively marked as `visited`, which seeds the v10.6 forward-looking
model. From that point on, the marker tracks correctly through the
remaining stops.

Activation requires moving at least 30m between fixes (rules out GPS
jitter while stationary).

## What this means for your bus 14 screenshot scenario

- **First fix** after panel opens: marker may briefly show "Approaching
  Bedok Camp 2" (the nearest stop, even if past). Same as v10.6.
- **Second fix, ~3 seconds later** (assuming the bus has moved ≥30m):
  direction inference identifies #16 as ahead and #15 as behind. Marker
  switches to "Approaching Opp Kew Green Condo" (#16). `lastVisitedSeq`
  seeds at 15.
- **Subsequent fixes**: forward-looking model takes over. As you pass
  #16, marker advances to #17, etc.

So instead of being stuck on the wrong stop indefinitely, the marker
self-corrects within ~6 seconds of opening the panel.

## Edge cases handled

- **Stationary at a stop**: no direction info. Falls back to nearest-stop
  highlighting, but doesn't seed `lastVisitedSeq`. Marker stable.
- **GPS jitter while stationary**: 30m minimum movement threshold blocks
  fake "movement" from being trusted as direction.
- **First-time fix**: no `prevFix` yet, falls back to nearest-stop.
  Self-corrects on the next fix when comparison is possible.
- **Bus stopped between stops**: similar to stationary; marker stays on
  whichever stop was identified as ahead, doesn't drift.
- **Bus turning**: each new fix updates `prevFix` (when ≥30m moved), so
  the direction inference uses recent heading, not stale.

## Edge cases that still misbehave (small)

- **First 3-6 seconds after opening the panel on a moving bus**: marker
  may briefly show the nearest stop, which could be a passed stop.
  Self-corrects on next fix.
- **Bus stationary at a red light between two adjacent stops**: marker
  may oscillate slightly between them as GPS jitters back and forth.
