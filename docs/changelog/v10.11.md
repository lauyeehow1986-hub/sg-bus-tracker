# v10.11 — Alighting alarm

## What's new

Tap the bell icon (🔕) next to any stop in the route list to set an
alighting alarm for it. The icon turns gold (🔔) and pulses to confirm.

When you're approaching the stop, the app fires:

- **Vibration burst** (Android haptic)
- **Audio chime** (WebAudio sine-wave beep, no audio file)
- **Spoken stop name** via SpeechSynthesis
- **System notification** (if you grant permission on first use)

Two alerts:

1. **At ~250m** — "Approaching Blk 55." Single beep, gentle vibration.
2. **At ~80m** — "Arriving at Blk 55. Press the bell now." Double beep,
   stronger vibration. Auto-disarms 5 seconds later.

## Honest limitations

This works **only while the app is in the foreground** and the screen
is on. PWAs can't reliably run background code on Android with the
phone locked. To compensate, arming the alarm acquires a Screen Wake
Lock so the screen stays on for the duration of the journey.

The arm-alarm toast says "Keep app open & screen on" so expectations
are clear.

If you specifically need a "phone in pocket / screen off" experience,
SG BusLeh has it (native Android app with proper background services).
This app's alarm is for the "I'm reading on the bus, screen face-up"
case.

## Speech: abbreviation expansion

Bus stop names are full of abbreviations that TTS reads literally
("Blk 55 Bedok Stn Exit B" → "B L K 55 Bedok S T N Exit B"). The
alarm expands ~25 common ones before speaking:

  Blk → Block, Stn → Station, Opp → Opposite, Aft → After,
  Ind → Industrial, Cplx → Complex, Pk → Park, Rd → Road,
  Ave → Avenue, Cres → Crescent, Sec → Secondary, Sch → School,
  Int → Interchange, etc.

Pronunciation of Singapore place names ("Bedok", "Bishan", "Tanah
Merah") still relies on the default voice and may sound mangled. We
accept this; the user knows what stop they meant.

## What happens if permission is denied

- **Notification denied**: vibration + chime + speech still work
- **Vibration unavailable** (some Android settings disable for browser):
  silent
- **Speech unavailable**: silent
- **Wake-lock unavailable** (older browsers): screen may sleep, alarm
  pauses until you wake it manually

Each capability degrades independently. The alarm always tries to fire
the alerts it can.

## UI

- **🔕 button** on every stop row (right-aligned next to the seq number)
- **🔔** when armed (gold, pulsing)
- Tap to toggle arm/disarm
- Closing the route panel automatically disarms

Only one alarm at a time — arming a new stop replaces the previous.

## Edge cases

- **Direction switch while armed**: alarm stays armed (the destination
  stop is geographically the same), but the bell may disappear from the
  visible list if the stop isn't on the new direction. Switch back to
  the original tab to disarm via the bell.
- **Auto-direction-switch (v10.3) while armed**: same as above — the
  alarm doesn't care which direction tab is active, it just watches
  distance.
- **Phone screen turns off anyway**: wake-lock handles this. If wake-lock
  fails (not supported, OS killed), the alarm pauses; you'll get the
  alert when you next wake the phone.
- **First speech call may be silent**: we play a silent test utterance
  on arming so subsequent speak() calls have a registered user gesture.
